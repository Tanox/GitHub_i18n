# 代码审查快速检查清单 (Quick Checklist)

> 💡 **使用说明**: 打印此清单或保持在旁边，审查代码时逐项检查

---

## 🔴 Blockers (必须修复)

### 正确性

- [ ] 逻辑是否正确实现了预期功能？
- [ ] 是否处理了边缘情况（null、undefined、空数组）？
- [ ] 异步代码（Promise/async-await）是否正确处理错误？
- [ ] 是否有明显的 off-by-one 错误？

### 安全性

- [ ] 是否避免了 XSS 风险（innerHTML、eval）？
- [ ] 是否验证了用户输入？
- [ ] 是否意外提交了敏感信息（API Key、密码）？
- [ ] 是否进行了适当的权限检查？

### 可靠性

- [ ] 是否有未捕获的异常？
- [ ] 是否有内存泄漏风险（未清理的事件监听器、定时器）？
- [ ] 是否有无限循环/递归风险？

---

## 🟡 Suggestions (建议修复)

### 可维护性

- [ ] 函数是否过长（建议 < 50 行）？
- [ ] 文件是否过大（建议 < 300 行）？
- [ ] 变量/函数命名是否清晰？
- [ ] 是否有适当的注释（复杂逻辑）？
- [ ] 是否有重复代码可以提取？

### 性能

- [ ] 是否有不必要的重复计算？
- [ ] 是否有 N+1 查询问题？
- [ ] DOM 操作是否高效（使用 DocumentFragment）？
- [ ] 是否可以适当使用缓存？

### 测试

- [ ] 核心功能是否有测试覆盖？
- [ ] Bug 修复是否有回归测试？
- [ ] 测试是否可靠？

---

## 💭 Nits (锦上添花)

- [ ] 代码风格是否一致？
- [ ] 是否有多余的注释？
- [ ] 是否有可以改进的命名（个人偏好）？

---

## 📋 项目特定检查点

### 用户脚本特殊检查

- [ ] 是否兼容目标浏览器版本？
- [ ] 是否会显著拖慢页面加载？
- [ ] 长期运行是否会占用过多内存？
- [ ] @match 规则是否正确？
- [ ] @grant 权限是否必要？

### 构建和工具

- [ ] 是否通过了 `npm run lint`？
- [ ] 是否通过了 `npm run build`？
- [ ] 是否通过了 `npm run test`？

---

## 💬 审查意见模板

### Blocker 问题

```markdown
🔴 **[类别]: [问题标题]**
Line X: [具体描述问题]

**Why:** [解释为什么这是问题]

**Suggestion:**
[提供具体的修复建议和代码示例]
```

### Suggestion 问题

```markdown
🟡 **[类别]: [问题标题]**
Line X: [具体描述问题]

**Why:** [解释为什么建议修改]

**Suggestion:**
[提供改进建议]
```

### Nit 问题

```markdown
💭 **Nit: [问题标题]**
Line X: [描述]

**Note:** This is a minor suggestion, feel free to ignore.
```

### 表扬

```markdown
👍 **Great job!**
[具体说明哪里写得好]
```

---

## ⚡ 快速审查技巧

1. **先看整体** — 理解 PR 的目的和实现思路
2. **再看细节** — 逐行检查代码质量
3. **运行测试** — 确保功能正常（如果可能）
4. **保持友善** — 假设作者善意，友好地提出建议
5. **及时响应** — 不要拖延审查

---

## 🚫 不要做的事项

- ❌ 不要只说"这不对"，要解释为什么
- ❌ 不要进行人身攻击或贬低
- ❌ 不要纠结于个人风格偏好（如果项目有统一规范）
- ❌ 不要 Block 只有 Nit 级别问题的 PR
- ❌ 不要在不理解的情况下随意批准

---

## ✅ 最终检查

在批准 PR 前，确认：

- [ ] 所有 🔴 Blocker 问题已修复
- [ ] 所有 🟡 Suggestion 问题已讨论
- [ ] PR 描述清晰，理解了改动目的
- [ ] 代码质量符合要求
- [ ] 没有影响其他功能

---

**记住**: 代码审查是为了提高代码质量和帮助团队成长，不是批评或指责！ 🚀
