INTERACTION · ERRORS

错误处理规范

错误状态下提供清晰、可操作的信息,而非技术化的提示。

表单错误 · Form Validation Error

Input-level · Inline message

字段级错误应紧邻输入框显示,红色边框强调问题位置,下方提供明确的修改建议。

词典路径 *
✗ 词典路径不存在
提示:请检查路径拼写或先在该位置创建 JSON 文件
GitHub 个人令牌
✗ 令牌无效或已过期
提示:在 GitHub Settings → Developer settings → Personal access tokens 中生成新令牌

网络错误 · Network Failure

Full-page / panel state block

当用户的设备无法连接到服务器时,提供一个居中的状态块,说明当前情况,并给出"重试"按钮。

⟳

网络请求超时

无法连接到更新服务器,请检查网络连接后重试。当前翻译内容不会丢失。

权限错误 · Permission Denied

Auth / Access level

当用户尝试执行无权操作时,不要只说"403"。告知他们缺少什么权限、由谁授予,以及如何联系管理员。

🛡

无操作权限

当前账号无法访问该仓库的设置。建议检查 GitHub 令牌的 repo 权限范围,或联系仓库所有者。

数据错误 · Data / Parse Error

Invalid format · Corrupt content

当用户提供的 JSON 文件格式不正确时,给出具体的错误位置,并提供"在编辑器中打开"的快捷入口。

⚠

词典格式无效

JSON 第 12 行解析失败,可能缺少逗号或引号不匹配。已保留原始内容,请修正后重新加载。

11: "commit": "提交",
12: "issue" "议题" ← 缺少冒号
13: "pull request": "拉取请求",

全局错误 · Error Boundary

Top-level alert banner

当页面发生未捕获的运行时错误时,在页面顶部显示一条深色横幅,告知用户"已自动恢复",并提供查看日志的入口。

⚠ 页面发生异常,已自动恢复翻译状态,不影响您保存的词典内容。 查看日志 →
ⓘ 检测到 3 个词条翻译与社区词典冲突,已保留您的本地版本。 查看差异 →

使用规则 · Design Guidelines

Principles for designing good error states

  • 不说"发生错误":替换为"翻译失败"、"连接超时"等具体描述,让用户知道发生了什么而非抽象的"错误"。
  • 给出下一步:每条错误消息后都应提供可点击的操作——重试、修改、查看文档、联系支持——而不是只让用户"知道"。
  • 保护已有数据:无论什么错误,已保存的用户数据必须不丢失。页面崩溃时,使用 Error Boundary 恢复到上一个稳定状态。
  • 使用技术码做诊断,不做 UI 主信息:HTTP 403、JSON parse error 等代码放在辅助行或日志详情中,标题保持人类可读。
  • 错误严重性层次:字段级错误→红边框+内联提示;页面级错误→状态块;全局错误→横幅。不要用全屏弹窗吓唬用户。
  • 色彩语义严格:错误类使用红色,权限类使用橙色(警告),信息类使用蓝色,不可混用以免用户对严重程度产生误判。
